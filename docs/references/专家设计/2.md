一个完整的本地股票数据存储方案，核心思路是**分层生命周期管理**（Hot-Warm-Cold-Archive）+ **自动过期清理**。

## 一、数据分级体系

### 1. 按时效性分级

| 级别 | 数据类型 | 保留策略 | 存储介质 |
|------|---------|---------|---------|
| **Hot** | 实时行情、当日分时 | 1天 | 内存/SSD |
| **Warm** | 近期K线（1月内）、持仓明细 | 动态保留（持仓期间+30天） | SSD |
| **Cold** | 历史K线（1年）、财务数据 | 固定保留期（3年） | SSD/HDD |
| **Archive** | 过期K线、已清仓标的 | 长期归档（10年+） | HDD/对象存储 |

## 二、存储引擎设计

### 1. 多层存储架构

**内存层（Redis/LRU Cache）**
- 存储：当日实时数据、活跃股票列表、用户当前持仓
- 清理：进程重启即清，或每日收盘后清空

**本地数据库（SQLite/LevelDB/RocksDB）**
- 近期数据（3个月内）
- 高频查询数据（自选股票池）
- 支持TTL（Time To Live）的数据库特性

**文件系统（Parquet/CSV压缩包）**
- 历史数据按月/年打包
- 使用**列式存储**（Parquet）+ Snappy压缩，查询比CSV快10倍，体积小5倍

**归档存储（可选）**
- 超过3年的数据打包为`.tar.zst`移至廉价存储
- 提供"按需解压"接口，查询时自动解压到临时目录

## 三、过期清理策略

### 1. 渐进式清理（推荐）

不要一次性删除大量数据，采用**渐进式清理**避免IO阻塞：

```python
class DataLifecycleManager:
    def __init__(self, db_path):
        self.db = db_path
        self.batch_size = 1000  # 每批清理1000条
        
    def daily_cleanup(self):
        """每日凌晨执行"""
        categories = [
            ("tick_data", 7),
            ("minute_kline", 90), 
            ("daily_kline", 1095)  # 3年
        ]
        
        for data_type, days in categories:
            self._gradual_delete(data_type, days)
            
    def _gradual_delete(self, data_type, retain_days):
        """分批删除，避免锁表"""
        cutoff_date = datetime.now() - timedelta(days=retain_days)
        
        while True:
            # 每次只删1000条
            affected = self.db.execute(f"""
                DELETE FROM stock_prices 
                WHERE data_type = ? AND timestamp < ?
                LIMIT {self.batch_size}
            """, (data_type, cutoff_date)).rowcount
            
            if affected == 0:
                break
            time.sleep(0.1)  # 给磁盘喘息时间，避免影响实时查询
```

## 四、容量监控与预警

### 1. 存储水位监控

```python
class StorageMonitor:
    def check_storage(self):
        usage = {
            'hot_data': self.get_folder_size('./hot/'),
            'warm_data': self.get_folder_size('./warm/'),
            'cold_data': self.get_folder_size('./cold/'),
```

## 五、关键优化技巧

### 1. 数据去重与采样
- **Tick数据**：盘后只保留**1分钟采样**，原始Tick7天后删除（体积减少99%）
- **停牌股票**：连续30天无成交的数据标记为`inactive`，迁移到慢速存储

### 2. 差分存储
存储每日增量而非全量：
- 日K线：永久保留（每天1条，体积很小）
- 分时数据：保留近期完整数据，历史只存**高低开收+成交量**（放弃明细）

### 3. 压缩策略矩阵

| 数据类型 | 压缩算法 | 压缩率 | 查询速度 | 备注 |
|---------|---------|-------|---------|------|
| Tick数据 | Zstandard | 5:1 | 快 | 平衡型 |
| K线数据 | Snappy | 2:1 | 极快 | 速度优先 |
| 归档数据 | LZMA2 | 10:1 | 慢 | 体积优先 |

### 4. 元数据索引
维护一个`data_manifest.json`，记录：
- 每个股票代码的数据时间范围
- 数据存储位置（数据库/文件/归档）
- 最后更新时间
- 校验和

避免查询时全盘扫描。

## 六、实施路线图

**Phase 1（基础）**：实现数据写入时自动打标签（expire_at），每日定时清理脚本
**Phase 2（优化）**：实现冷热分离，3个月以上数据自动转Parquet
**Phase 3（高级）**：实现压缩归档、云端备份接口、存储预测算法

这个方案的核心是**"写入时标记过期时间，查询时忽略过期数据，后台渐进清理"**，既能保证查询性能，又能防止磁盘无限增长。
