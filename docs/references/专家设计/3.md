股票分析软件的本地数据存储规划，核心痛点就是**时序数据的高频写入与无限膨胀**。为了解决这个问题，我们需要从“数据分类”、“冷热分离”、“降采样（聚合）”和“存储引擎选型”四个维度来设计一套完整的生命周期管理方案。
以下是为你规划的本地数据存储机制：
### 一、 数据分层与时效性定义
首先要明确，不是所有数据都需要一样的保质期。我们将数据分为三类：
1.  **基础面/静态数据（全量保留，无限期）**
    *   *内容：* 股票列表、代码映射、行业板块、财务报表（季报/年报）、除权除息信息。
    *   *特点：* 更新频率低，数据量极小，几十MB级别，不需要清理。
2.  **日K线及以上级别（全量保留，无限期）**
    *   *内容：* 日线、周线、月线。
    *   *特点：* 假设有5000只股票，保留20年数据，每天一条，总计约 `5000 * 20 * 250 = 2500万条`。对现代数据库来说这几乎不占空间（不到1GB），可以永久保留。
3.  **分钟K线 / Tick逐笔数据（滑动窗口，定期清理）**
    *   *内容：* 1分/5分/15分/30分/60分钟K线，Level-2 Tick数据。
    *   *特点：* **数据爆炸的元凶**。1分钟K线一天就有 `5000 * 240 = 120万条`。这类数据是清理的核心目标。
---
### 二、 滑动窗口与降采样机制（控制增长的核心）
针对分钟级和Tick数据，采用**“保留近期明细 + 历史数据降采样”**的策略：
*   **Tick数据（逐笔）：** 仅保留**最近 N 个交易日**（例如30天）。超过30天的Tick数据直接删除。如果用户需要回测，必须将Tick聚合为1分钟K线后再保留。
*   **1分钟/5分钟K线：** 保留**最近 1~2 年**的原始数据。超过2年的数据，执行降采样：将其聚合为日线后删除原分钟数据（因为日线上面已经有了，不需要在分钟表里存两份）。
*   **15/30/60分钟K线：** 可以永久保留，因为数据量已经被大幅过滤了（一天只有几条），或者同样保留2年后降采样为日线。
**💡 降采样校验原则：**
清理前必须确保高精度数据已经正确生成了低精度数据。例如：删除某天的1分钟数据前，检查该天的日K线（开高低收、成交量）是否已经存在且正确。
---
### 三、 存储引擎与文件组织建议
强烈建议**不要把所有数据塞进一个单一的SQLite或MySQL里**，这会导致后期清理数据时产生大量碎片，甚至锁表。推荐采用 **“元数据用关系型DB + 行情数据用文件化存储”** 的混合架构：
#### 推荐方案：SQLite（存配置） + DuckDB 或 Parquet文件（存行情）
*(DuckDB是目前本地量化分析的神器，专门针对列式时序数据优化，且支持直接查询Parquet文件)*
**目录结构设计：**
```text
/data
  /meta_db.sqlite              # 存基础信息、用户配置、自选股等
  /daily                       # 日K线 (永久保留)
    /000001.parquet            # 按单只股票一个文件，或者按年份分片
  /minute                      # 分钟K线 (滑动窗口)
    /2023                      # 按年份分目录
      /000001_2023.parquet
    /2024
      /000001_2024.parquet
  /tick                        # 逐笔数据 (极短滑动窗口)
    /202410                     # 按月份分目录
      /000001_202410.parquet
```
**为什么这样设计？**
1. **清理极其简单：** 要删除2022年的1分钟数据，直接删除 `/minute/2022` 整个文件夹即可，没有数据库碎片问题。
2. **查询性能高：** Parquet是列式存储，读取某只股票最近一年的 OHLC（开高低收），速度比传统行式数据库快数倍。
3. **防膨胀彻底：** 文件级别的管理，说删就删，磁盘空间立刻释放。
---
### 四、 数据生命周期调度任务
你需要在软件后台（或者作为一个独立的守护进程）设定一个定时任务，通常在**每天收盘数据更新完成后（例如下午16:00）**执行：
**每日清理与聚合流水线：**
1.  **数据入库与校验：** 拉取今天的Tick和分钟数据，写入对应的 Parquet/数据库。
2.  **向上聚合（自底向上）：** 
    *   用今天的 Tick 聚合出今天的 1分钟K线。
    *   用今天的 1分钟K线 聚合出今天的 日K线。
    *   更新财务/除权等静态数据。
3.  **过期数据清理（自顶向下）：**
    *   扫描 `/tick` 目录，删除30天前的月份文件夹。
    *   扫描 `/minute` 目录，删除2年前的年份文件夹。
4.  **一致性巡检（可选）：** 抽查几只股票，验证 `日K线的成交量 == 当天所有分钟K线成交量之和`，防止数据丢失。
---
### 五、 补充：软件使用体验的优化（冷热分离）
如果按照上述方案，当用户在软件里看“某只股票3年前的1分钟图”时，发现没数据怎么办？需要做好前端交互的冷热分离：
1.  **热数据（内存/本地磁盘）：** 近期数据秒开。
2.  **冷数据提示：** 当用户拖动K线图时间轴，请求范围超过了本地保留的2年界限时，UI应当拦截并弹窗提示：“*当前查看时间超出本地存储范围（本地仅保留近2年分钟数据），是否从服务器重新下载该时段数据？*”
3.  **按需拉取（懒加载）：** 用户点击“是”后，临时从你的数据源下载该片段的历史分钟数据，仅用于当前显示，**用完即弃或者放在缓存目录（Cache）中，不进入核心存储库**，下次软件重启依然可以清理掉。
### 总结
这套方案的精髓在于：**静态数据和日K无脑存，分钟/Tick设期限，通过文件级（Parquet）管理替代传统的数据库Delete操作，用定时聚合保障历史回测的底线需求。** 这样你的软件即使跑上几年，本地占用的空间也能死死锁定在一个可控的范围内（通常在几个GB到十几个GB以内，取决于Tick保留时长）。
