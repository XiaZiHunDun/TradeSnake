针对股票分析软件的本地数据存储，核心目标是**控制数据总量**、**保证时效性**、**兼顾查询性能**。以下方案从数据分类、存储架构、生命周期管理三个维度设计，并提供可落地的清理策略。

---

## 一、数据分类与保留策略

将本地数据分为三类，分别设定保留规则：

| 数据类型 | 示例 | 默认保留时长 | 最大记录数 | 说明 |
|---------|------|-------------|-----------|------|
| **高频实时数据** | 1分钟K线、分时Tick | 30天 | 每只股票最多10万条 | 时效性最强，快速膨胀，需频繁清理 |
| **中频分析数据** | 日K线、周K线、月K线 | 5年 | 无限制（按时间截断） | 数据量可控，长期保留用于回测 |
| **低频元数据** | 股票列表、除权除息、财报日期 | 永久 | 无限制 | 数据量小，一般不清理，仅更新 |

> 用户可自定义每类数据的保留期限或最大存储空间（如总库上限1GB）。

---

## 二、本地存储架构

采用 **SQLite + 分表 + 时间分区** 的组合：

1. **单库多表**：每个股票一个表，表名 `stock_{code}_{period}`（如 `stock_000001_day`），避免全表扫描。
2. **时间分区**（SQLite 3.9+ 支持）：按月份/年份创建虚拟表或手动分表（如 `kline_1min_2025_01`），便于批量删除。
3. **索引**：`(date, code)` 复合索引，加速时效性查询。
4. **元数据表**：`data_metadata` 记录每张表的最后更新时间、记录数、数据范围，供清理逻辑使用。

示例建表（日线）：
```sql
CREATE TABLE stock_000001_day (
    date TEXT PRIMARY KEY,   -- YYYY-MM-DD
    open REAL, high REAL, low REAL, close REAL,
    volume INTEGER, amount REAL
);
```

## 三、数据生命周期管理（核心机制）

### 3.1 写入时的检查

每次新增数据前，先查询当前表的记录数或最早日期：
- 若超出“最大记录数”或“保留天数”，触发**滑动窗口删除**。
- 滑动窗口删除：例如日线保留5年，每次写入新数据后，删除 `date < date('now', '-5 years')` 的记录。

### 3.2 定期清理任务（后台线程）

每天收盘后或软件启动空闲时执行：

1. **遍历所有数据表**，根据表名后缀识别周期类型。
2. **执行删除**：
   - 高频数据：`DELETE FROM stock_{code}_1min WHERE date < datetime('now', '-30 days')`
   - 中频数据：`DELETE FROM stock_{code}_day WHERE date < date('now', '-5 years')`
3. **VACUUM** 回收空间（可每周一次，或当删除量超过总大小20%时触发）。
4. **更新元数据表**：记录清理后的最小/最大日期。

### 3.3 存储空间上限保护（安全网）

设置总数据库文件大小上限（如2GB）。当接近上限（95%）时：
- 强制缩短所有数据的保留期限（例如将1分钟线从30天降为15天）。
- 删除最旧的未使用数据（按最后访问时间）。
- 若仍不足，发出用户警告并停止写入新数据。

### 3.4 时效性数据更新策略

- **增量更新**：每次请求API时，只拉取最新日期的数据，避免全量下载。
- **本地去重**：写入前检查 `date` 是否已存在，存在则更新（除权修正场景），避免重复记录。
- **数据补全**：用户选择较长周期时（如5年），检查本地是否缺漏中间时间段，自动分批补拉。

## 五、异常与容错

1. **清理失败**：若删除时数据库繁忙（如正在查询），重试3次，间隔1秒；若仍失败，记录日志，下次启动时再清理。
2. **数据空洞**：因清理导致中间数据缺失（用户突然缩短保留期），下次增量更新时会自动跳过已删除部分，无需特殊处理。
3. **多设备同步**（可选）：若软件支持云同步，本地只缓存最近3个月数据，完整历史保存在云端。本地清理时不删除云端的全量数据。

## 六、示例代码逻辑（伪代码）

```python
class DataCleaner:
    def __init__(self, db_path, config):
        self.db = sqlite3.connect(db_path)
        self.config = config  # 包含每种周期的保留天数/记录数限制

    def clean_table(self, table_name, period_type):
        if period_type == '1min':
            keep_days = self.config['keep_1min_days']
            self.db.execute(f"DELETE FROM {table_name} WHERE date < datetime('now', '-{keep_days} days')")
        elif period_type == 'day':
            keep_years = self.config['keep_day_years']
            self.db.execute(f"DELETE FROM {table_name} WHERE date < date('now', '-{keep_years} years')")
        # 可选：同时检查记录数，若超过max_rows则删除最旧的超出部分
        affected = self.db.total_changes
        if affected > 0:
            self.db.execute(f"VACUUM")   # 可改为异步或定时
        return affected

    def run_daily_clean(self):
        tables = self.get_all_stock_tables()
        for table in tables:
            period = self.detect_period(table)
            self.clean_table(table, period)
```

## 七、总结优势

- **数据量可控**：通过保留期限和空间上限双重保险，避免无限增长。
- **查询性能**：定期清理 + 索引 + 分表，保证即使多年使用也不卡顿。
- **用户自主**：提供配置界面，满足不同分析需求（短线/长线）。
- **低维护成本**：依赖SQLite内建机制，无需额外服务进程。

此方案可直接集成到你的股票分析软件中，如需更具体的SQL语句或清理触发器设计，可进一步细化。
