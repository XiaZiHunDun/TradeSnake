# 历史股票数据接入方案

为股票分析软件提供历史股票数据，本质上是一个数据源的选型、获取、存储和更新的系统工程。以下从数据源对比选型、获取架构、存储方案、更新策略及合规建议五个方面，提供完整的方案。


## 一、数据源选型

当前市场主流的数据源可分为开源免费、商业付费和直接爬取三类，各有优劣。

### 1.1 开源数据接口对比

| 特性 | AkShare | Tushare | BaoStock | TickFlow |
|------|---------|---------|----------|----------|
| 价格 | 完全免费 | 积分制/低额付费 | 完全免费 | 免费套餐+付费升级 |
| 是否需要注册 | 无需 | 需要Token | 需要Token | 免费用户无需 |
| 数据覆盖 | A股、港股、美股、期货、外汇、宏观数据 | A股为主，覆盖港股、美股 | 仅A股 | A股、港股、美股、ETF、期货 |
| 历史数据深度 | 全量历史 | 全量历史 | 全量历史 | 全量历史 |
| 复权支持 | 依赖数据源 | 完整 | 标准复权数据 | 等差/等比复权 |
| 数据稳定性 | ❌ 依赖源站，易中断 | ✅ 稳定可靠 | ✅ 稳定可靠 | ✅ 可靠 |
| 实时行情 | 依赖源站，不可靠 | 需高积分 | ❌ 不支持 | 支持（需付费） |
| 分钟级数据 | 仅近几天 | 需2000+积分 | 支持 | 近一年+实时 |
| 适用场景 | 多品种研究、原型开发 | A股量化首选 | A股历史回测 | 生产环境、全市场 |

**选型建议**：

- **初学者/个人研究者**：AkShare 开箱即用，数据覆盖广，适合快速上手和低成本研究[reference:0]。但需注意其接口依赖网页解析，稳定性是最大短板。

- **A股量化回测**：BaoStock 在历史数据的准确性和复权一致性方面表现突出，虽然不支持实时行情，但作为历史数据源非常可靠[reference:1]。

- **生产级A股应用**：Tushare Pro 数据质量高，API调用成功率超过99%，是构建量化策略的首选工具[reference:2]。缺点是免费版限制较多，分钟级数据需2000+积分，长期使用成本逐年上升[reference:3]。

- **生产级多市场应用**：TickFlow 提供稳定的免费套餐，支持全量历史K线和分钟级数据，性价比突出[reference:4]。

- **全球市场覆盖**：OpenBB 是一个开源金融数据平台，统一接口对接全球多数据源，支持Python、CLI、REST API等多种消费方式[reference:5]。yfinance 获取美股历史数据简单易用，但需注意Yahoo Finance数据源本身偶有错误且在国内网络环境下访问可能不稳定[reference:6]。

### 1.2 商业付费数据源

| 服务商 | 特点 | 参考费用 |
|--------|------|----------|
| 同花顺OpenAPI | 毫秒级实时行情、Level-2 K线 | 数万元/年 |
| Wind API | 全球市场全覆盖，专业级 | 年费高昂 |
| iTick | 港股美股突出，毫秒级延迟 | 基础免费+专业升级 |
| Polygon.io | 实时逐笔数据、WebSocket推送 | $29/月起 |
| Alpha Vantage | 内置技术指标，接口规范 | 免费+$29.99/月起 |
| EODHD | 全球清洁历史数据、基本面 | 免费+$79.99/月 |

对于早期阶段，免费开源方案已足够；若软件面向付费用户或需要高频数据支撑复杂策略，建议逐步接入商业服务。

### 1.3 东方财富与腾讯财经直接API

东方财富和腾讯财经提供可直接调用的免费API接口，可绕过中间库直接获取数据：

- **东方财富API**：历史数据周期长，复权数据支持好，适合长期历史回测[reference:7]。
- **腾讯财经API**：响应速度快，实时性高，适合实时监控场景[reference:8]。

直接调用原始API可减少中间层依赖，但同样面临接口变更和限流的风险。


## 二、数据获取与存储架构

### 2.1 推荐架构：多源备份 + 本地数据库

考虑到免费数据源随时可能变动或失效，强烈建议采用多源备份策略。TradingAgents项目的实践经验值得借鉴：

- **主数据源**：Tushare（稳定可靠）
- **备用数据源**：AkShare（自动无缝切换）
- **历史数据源**：BaoStock（补充使用）
- **优先层**：本地缓存数据库[reference:9]

数据获取流程图：

```
用户请求
    ↓
本地缓存数据库 ──有数据且未过期──→ 直接返回
    ↓ 无数据或已过期
主数据源（如 Tushare/TickFlow）
    ↓ 失败
备用数据源（如 AkShare/BaoStock）
    ↓
数据校验与清洗
    ↓
存入本地数据库
    ↓
返回给用户
```

### 2.2 存储方案选型

| 方案 | 适用场景 | 特点 |
|------|----------|------|
| **Parquet + 对象存储** | 中小规模、回测分析 | 列式存储、压缩率高、与Pandas/Polars无缝对接 |
| **SQLite / PostgreSQL** | 中小规模、结构化查询 | 成熟稳定、易于管理 |
| **TDengine / TimescaleDB** | 大规模时序数据、高频查询 | 专用时序数据库，查询性能极佳 |
| **ArcticDB** | Python生态、金融时间序列 | 专为时序数据优化，支持版本管理 |
| **ClickHouse** | 海量数据、实时分析 | 列式存储、查询极快 |

**推荐方案**：对于中小型股票分析软件，初期可采用 **Parquet + 对象存储** 组合，数据按股票代码和日期分区存储。Parquet列式存储配合Pandas/Polars可实现高效查询，压缩率可达7%左右[reference:10]，存储成本低且迁移灵活。

若未来数据量激增，可平滑迁移至专业时序数据库如TDengine或TimescaleDB。令克软件的实践表明，TDengine的“一条时间序列一张表”设计与金融场景高度契合，能够同时满足大规模写入与高频查询的性能要求[reference:11]。

**关键字段**：date、open、high、low、close、volume、adjust_factor（前复权因子）


## 三、数据更新策略

### 3.1 全量初始化

软件首次上线时，需要为关注的股票拉取全量历史数据。建议采用批量拉取 + 限流控制策略，避免被目标数据源封禁：

- 使用 `time.sleep()` 控制请求间隔（建议0.5-1秒/次）
- 设置请求超时和重试机制（最多3次重试）
- 优先拉取上证50、沪深300等核心标的，再扩展至全市场

### 3.2 增量更新

每日收盘后更新当日数据，可采用以下策略：

```
每日15:30触发增量更新
    ↓
获取已覆盖的股票列表
    ↓
调用API获取最近5个交易日数据
    ↓
与本地最新数据比对，去重合并
    ↓
更新本地存储，删除临时数据
```

**关键注意事项**：

- **复权处理**：技术分析需要前复权价格序列，务必使用复权因子对历史价格进行调整，消除分红、送股对股价的影响。
- **除权除息日调整**：在除权除息日附近，复权数据的获取容易出现错误，建议在增量更新时特别校验这些日期的数据。
- **数据完整性校验**：每次更新后检查OHLCV数据是否完整，成交量是否为正，价格是否在合理范围内。

### 3.3 定期全量校验

建议每月或每季度执行一次全量校验，将本地数据与官方数据源进行比对，修复可能存在的数据漂移问题。


## 四、合规与风险提示

### 4.1 法律合规

**这是整个方案中最需要重视的部分。**

2026年1月1日，修订后的《网络安全法》正式实施，对“未经授权获取网络数据”的定义进行了前所未有的收紧。直接解析网页、绕过反爬等手段，在法律定义上正在滑向“非法侵入”的灰色地带[reference:12]。

**合规建议**：

1. **优先使用官方授权的API**：如Tushare、TickFlow等需注册获取Token的服务，有明确的授权使用协议。
2. **避免直接爬取**：AkShare等依赖网页解析的库，虽然技术上可行，但合规风险日益增加。
3. **关注官方开放平台**：东方财富已推出官方Skills标准化技能服务，这是金融数据获取从“地下”走向“地上”的重要信号[reference:13]。
4. **个人使用 vs 商业分发**：若软件将面向第三方用户分发，需格外注意数据授权问题。直接集成AkShare等爬虫类接口存在法律风险，建议采用合法授权的商业数据源。

### 4.2 数据质量保障

免费数据源可能存在缺失值、异常值，建议在入库前进行数据清洗和校验。有行业人士特别指出：使用akshare和新浪财经数据时要注意数据质量，可能存在数据质量差或接口被封的问题[reference:14]。


## 五、实施建议与路线图

### 阶段一（MVP验证，1-2周）

- 选择 AkShare 或 TickFlow 免费套餐作为数据源
- 使用 Python + Parquet 存储本地数据
- 实现 200-500 只核心股票的日K线数据获取和展示
- 验证软件核心功能的可行性

### 阶段二（功能完善，2-4周）

- 扩展数据源覆盖范围至全部A股（约5000+只）
- 增加分钟级数据获取能力
- 实现数据增量更新机制
- 增加数据完整性校验和异常告警

### 阶段三（生产部署，4-8周）

- 接入双数据源备份机制（主源+备用源）
- 迁移至专业时序数据库（如需要）
- 引入商业数据源补充实时行情需求
- 完成合规审查和授权确认


## 总结

| 需求场景 | 推荐方案 | 成本 |
|----------|----------|------|
| 个人开发、快速验证 | AkShare + Parquet | 免费 |
| A股历史数据回测 | BaoStock/Tushare Pro + Parquet | 低 |
| 生产级A股应用 | TickFlow/Tushare Pro + TDengine | 中等 |
| 多市场覆盖 | OpenBB + 本地缓存 | 低-中等 |
| 商业产品分发 | 商业数据API + 专业时序数据库 | 较高 |

**一句话建议**：先用 AkShare 或 TickFlow 免费套餐跑通原型，再用 BaoStock 做历史数据补充，后期根据商业化程度接入 Tushare Pro 或商业数据API。核心原则：多源备份 + 本地存储 + 合规优先。
