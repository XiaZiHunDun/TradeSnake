# TradeSnake 数据生命周期管理方案 v1.1 评审意见
## 一、整体评价
该方案**结构完整、逻辑清晰**，覆盖了数据分级、保留策略、清理机制、实施计划等核心维度，贴合金融类应用数据管理的核心诉求（核心资产数据绝对保护+非核心数据高效清理），整体具备可落地性。以下从**优势亮点**、**待优化点**、**风险提示**、**落地建议**四个维度展开评审。

## 二、核心优势
### 1. 分级存储策略精准
- 按「P0-P5」分级贴合业务本质：P0（持仓/交易记录）永久保留符合金融场景用户资产数据不可丢失的核心诉求，P1（战力历史）永久保留兼顾业务分析需求，P2-P5按时效性梯度清理，平衡「数据价值」与「存储成本」。
- 分级与存储介质（SQLite/DuckDB/JSON）强绑定，清理策略与介质特性匹配（如DuckDB K线批量清理、JSON缓存按TTL清理）。

### 2. 清理机制兼顾安全与效率
- 「渐进式清理」设计规避IO阻塞风险：按「临时文件→缓存→备份→K线→SQLite历史」的优先级清理，先非核心后次要核心，且分「每日/每周」调度，避免单次清理压力过大。
- 存储水位保护机制（70%警告/80%强制清理/95%禁止写入）形成闭环，防止存储耗尽导致服务不可用。

### 3. 可配置化与用户体验平衡
- 预留用户可配置选项（日K保留年数、分钟K保留天数等），既满足标准化管理，又适配不同用户的个性化需求（如专业用户需更长K线数据回测）。
- 存储监控可视化设计（数据存储概览、健康状态提示），降低用户理解成本。

### 4. 落地性强
- 明确「Phase 1-3」实施计划，按「基础清理→监控告警→配置化」分步落地，节奏合理；
- 「关键文件对应」章节精准关联现有代码文件，避免方案与工程实现脱节；
- 速查表、专家设计对照等模块提升方案的可读性和追溯性。

## 三、待优化点
### 1. 核心数据保护的细节补充
- **P0/P1数据的备份冗余性**：方案仅提及「每日备份SQLite、每周备份战力历史」，未明确「多副本/异地备份」策略，若本地备份文件损坏，核心数据仍有丢失风险；
- **清理校验机制缺失**：未定义「清理前数据完整性校验」规则（如清理前校验cp_history表记录数是否正常、持仓数据是否无缺失），可能因代码BUG导致误删核心数据。

### 2. 清理机制的精细化不足
- **DuckDB日K线降采样的落地细节**：方案提及「降采样为月K归档」，但未明确：
  - 月K归档表（monthly_kline_archive）的结构定义；
  - 降采样后归档数据的查询兼容逻辑（如用户查询5年前日K时，自动路由到月K归档表）；
- **渐进式清理的「分批」粒度**：仅定义清理优先级，但未明确「单次清理的记录数上限」（如DuckDB单次清理不超过10万条），极端场景下仍可能引发IO阻塞；
- **缓存清理的「覆盖写」风险**：如stock_pool.json「更新时覆盖+7天清理」，若更新失败且旧文件被清理，可能导致配置丢失，需补充「覆盖前备份」逻辑。

### 3. 可配置化的边界约束
- 方案提及「用户可配置保留期限」，但未定义配置的「合法边界」（如日K保留年数最多10年，不可配置为「永久」，避免存储无限膨胀）；
- 未明确「配置修改后的生效规则」（如修改分钟K保留天数后，是立即清理还是等待下一次调度）。

### 4. 监控与告警的完整性
- 仅定义存储使用率告警，未覆盖「清理失败告警」（如某次DuckDB清理执行报错，需触发告警通知运维/用户）；
- 未提及「数据增长趋势监控」（如日K数据量月增长超50%，提前预警存储压力）。

### 5. 异常场景处理
- 未考虑「清理过程中服务重启」的情况：若清理到一半重启，可能导致数据不完整（如部分K线已删除、月K未归档），需补充「清理事务化」或「断点续跑」机制；
- 未定义「存储达到95%后，禁止新数据写入」的兜底方案（如用户无法下单/更新战力历史，需明确降级策略：仅禁止非核心数据写入，核心数据仍可更新）。

## 四、风险提示
### 1. 核心数据误删风险
- 清理逻辑若存在BUG（如SQL条件写错），可能误删P0/P1数据（如cp_history/holdings表），建议：
  - 所有清理SQL/代码添加「表名白名单校验」（仅允许清理price_history/alerts等非核心表）；
  - 清理前自动触发核心表备份，且备份文件标记为「清理保护副本」（7天内不可删除）。

### 2. 降采样数据精度损失
- 日K降采样为月K后，会丢失日内波动、单日成交量等细节，需在方案中明确「降采样仅针对5年前数据」，且保留「降采样前的最终确认环节」（如用户可选择是否执行降采样，而非自动执行）。

### 3. 配置化的运维成本
- 允许用户自定义保留期限后，可能出现「部分用户配置超长保留期导致存储耗尽」，需补充「用户级存储配额」（如单用户DuckDB存储上限2GB，超限后强制按默认策略清理）。

## 五、落地建议
### 1. 短期（Phase 1前）
- 补充「清理前核心数据校验+备份」逻辑，在daily_cleanup/weekly_cleanup函数中增加前置校验步骤：
  ```python
  def pre_cleanup_check():
      """清理前核心数据校验"""
      # 1. 校验P0/P1表记录数是否正常（无异常减少）
      core_tables = ['holdings', 'trades', 'cp_history']
      for table in core_tables:
          count = sqlite_db.query(f"SELECT COUNT(*) FROM {table}").scalar()
          if count < last_recorded_count[table] * 0.9:  # 记录数骤降10%告警
              raise Exception(f"核心表{table}记录数异常，终止清理")
      # 2. 自动备份核心表到独立保护目录
      backup_core_tables(protected_dir="./data/backup/protected/", expire_days=7)
  ```
- 明确DuckDB月K归档表结构，补充查询兼容逻辑：
  ```sql
  -- 月K归档表结构
  CREATE TABLE IF NOT EXISTS monthly_kline_archive (
      code TEXT NOT NULL,
      trade_month TEXT NOT NULL,  -- 格式：YYYY-MM
      open REAL NOT NULL,
      high REAL NOT NULL,
      low REAL NOT NULL,
      close REAL NOT NULL,
      volume BIGINT NOT NULL,
      created_at TEXT DEFAULT CURRENT_TIMESTAMP,
      PRIMARY KEY (code, trade_month)
  );
  ```

### 2. 中期（Phase 2前）
- 增加「清理失败告警」和「数据增长趋势监控」：
  - 清理任务执行后校验「实际清理记录数」与「预期清理记录数」，偏差超20%则触发告警；
  - 每日统计各存储介质的数据量，生成周/月增长趋势，若增速超阈值（如周增长10%）提前预警。
- 补充配置项的边界约束：
  ```python
  # 配置项合法值约束
  CONFIG_CONSTRAINTS = {
      'daily_kline_years': {'min': 1, 'max': 10},
      'minute_kline_days': {'min': 7, 'max': 180},
      'cache_ttl_days': {'min': 1, 'max': 30}
  }
  # 配置修改生效规则：立即生效，且触发一次增量清理
  def apply_config_change(new_config):
      validate_config(new_config, CONFIG_CONSTRAINTS)
      save_config(new_config)
      trigger_incremental_cleanup()  # 按新配置清理超期数据
  ```

### 3. 长期（Phase 3后）
- 落地「用户级存储配额」，在存储监控中按用户维度统计占用，超限后仅禁止非核心数据写入；
- 补充「清理事务化」机制，对DuckDB/SQLite清理操作封装事务，避免中途重启导致数据不完整：
  ```python
  def cleanup_old_daily_klines(keep_years: int = 5) -> Dict:
      with duckdb_store.transaction():  # 封装事务
          # 1. 降采样归档
          archive_to_monthly_kline(keep_years)
          # 2. 删除超期日K
          delete_old_daily_kline(keep_years)
      # 事务提交后更新清理记录
      update_cleanup_log('daily_kline', keep_years)
  ```

## 六、总结
该方案已具备「数据分级-保留策略-清理执行-落地计划」的完整闭环，核心优势是贴合业务场景、安全兜底机制完善。需重点补充**核心数据保护细节、清理精细化规则、异常场景处理**三类内容，即可大幅提升方案的健壮性和可落地性。

建议优先落地Phase 1的基础清理功能（补充校验和备份逻辑），再逐步推进监控告警和配置化，确保每一步落地都能兼顾「数据安全」与「存储效率」。
