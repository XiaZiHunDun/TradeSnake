## 二、具体问题与建议

### 2.1 状态管理设计

**问题：`kline_fill_status` 表设计过于复杂**

```sql
-- 当前设计：每个股票+日期范围一条记录
UNIQUE(code, start_date)
```

**建议简化：**
```sql
CREATE TABLE kline_fill_status (
    code VARCHAR(6) PRIMARY KEY,    -- 每只股票一条记录
    latest_date DATE,                 -- 最后填充日期（断点续跑用）
    status VARCHAR(20),               -- pending/running/completed
    updated_at TIMESTAMP
);
```

**理由：**
- 单股历史数据是连续的，不需要记录多个区间
- 简化后 `GapDetector` 直接查询 `daily_kline` 表即可，无需维护复杂状态
- 你的 `GapDetector.get_existing_range()` 已经能检测缺口，状态表只需记录"最后处理到哪"

### 2.2 GapDetector 与 FillerState 职责重叠

**现状：** `GapDetector` 查 DuckDB 找缺口，`FillerState` 又维护一套状态

**建议合并：**
```python
class KlineFiller:

        target_start = today - timedelta(days=days)
        
        # 计算实际需要填充的区间
        if existing_end:
            fill_start = existing_end + timedelta(days=1)
        else:
            fill_start = target_start
        
        # 填充并记录最新日期到SQLite（仅用于断点续跑）
```

**简化后的状态表只存：**
| code | latest_date | updated_at |
|------|-------------|------------|
| 000001 | 2026-04-08 | 2026-04-09 |

### 2.3 复权因子字段设计

**当前设计：**
```sql
adj_factor DECIMAL(10, 6) DEFAULT 1.0,    -- 前复权因子
adj_close DECIMAL(10, 2) DEFAULT 0,       -- 前复权收盘价
```

**建议改为：**
```sql
adj_factor DECIMAL(10, 6),        -- 复权因子（1.0表示无除权）
is_suspended BOOLEAN DEFAULT FALSE  -- 停牌标记（替代volume=0判断）
```

**理由：**
- `adj_close` 可以实时计算：`close * adj_factor`，无需存储
- 存储 `adj_factor` 即可支持前复权/后复权切换（后复权 = close × 最新因子 / 当日因子）
- 停牌标记比判断 `volume=0` 更可靠（有些股票成交量极小但非停牌）

### 2.4 多源备份策略的实现细节

**当前设计：** `MultiSourceProvider` 自动切换

**潜在问题：**
- 不同数据源的数据格式、复权方式可能不一致
- 混用数据源会导致同一股票历史数据不连贯

**建议：**
```python
class MultiSourceProvider:
    def get_daily_kline(self, code: str, start: str, end: str):
        # 策略：单只股票全程使用同一数据源
        # 记录该股票上次使用的数据源，优先复用
        
        preferred_source = self._get_preferred_source(code)
        data = preferred_source.get_daily_kline(code, start, end)
        
        if not data:
            # 只有全程失败才切换，且切换后该股票后续全部使用该源
            for source in self.providers:
                if source != preferred_source:
                    data = source.get_daily_kline(code, start, end)
                    if data:
                        self._set_preferred_source(code, source)
                        break
```

### 2.5 增量填充逻辑的边界情况

**当前代码：**
```python
def get_fill_start_date(code: str, ...):
    latest_date = db.query("SELECT MAX(trade_date) FROM daily_kline WHERE code = '{code}'")
    return (latest_date + timedelta(days=1)).strftime('%Y%m%d')
```

**问题：**
- 如果 `latest_date` 是今天，会请求明天的数据（无效）
- 如果股票今天停牌，会漏掉今天的0成交量记录

**建议：**
```python
def get_fill_range(code: str, days: int) -> Tuple[str, str]:
    """返回 (start_date, end_date)，确保覆盖所有交易日"""
    today = datetime.today().date()
    
    # 获取数据库最新日期
    result = db.query(f"SELECT MAX(trade_date) FROM daily_kline WHERE code = '{code}'")
    db_latest = result.iloc[0, 0] if result.success else None
    
    # 获取目标起始日期（考虑保留期限）
    retention_start = today - timedelta(days=days)
    
    if db_latest is None:
        # 全新股票：填充完整历史
        return (retention_start.strftime('%Y%m%d'), today.strftime('%Y%m%d'))
    
    if db_latest >= today:
        # 已是最新，无需填充
        return (None, None)
    
    # 增量：从最新日期的下一天开始
    fill_start = db_latest + timedelta(days=1)
    
    # 但如果最新日期早于保留期限，需要补中间缺失的
    if db_latest < retention_start:
        fill_start = retention_start
    
    return (fill_start.strftime('%Y%m%d'), today.strftime('%Y%m%d'))
```

### 2.6 数据对齐校验的实现成本

**建议暂缓实现 `DataAlignmentValidator`**

**理由：**
- 获取"某日期附近所有股票的交易日列表"需要全表扫描，性能开销大
- 实际场景：个别股票额外交易日通常是新股上市、复牌等正常情况
- 建议改为：填充时检查 `trade_date` 是否为A股交易日（用交易日历），而非与其他股票对齐

### 2.7 退市股处理的实际困难

**现状：** 方案提到"必须包含退市股"，但实现细节不足

**实际问题：**
- Tushare 的 `stock_basic` 默认只返回上市股票，退市股票需要特殊接口
- 退市股的历史数据获取接口可能不同

def get_stock_list(include_delisted: bool = True) -> List[Dict]:

### 2.8 优先级调整建议

| 原优先级 | 内容 | 建议调整 | 理由 |
|---------|------|---------|------|
| P1 | `AdjFactorHandler` 复权因子 | **P0** | 不复权的价格数据无法用于分析 |
| P1 | `DataValidator` 数据校验 | **P0** | 数据质量是核心，应在填充时实时校验 |
| P2 | `MultiSourceProvider` 多源备份 | P2 | 生产环境建议先用单一稳定源 |
| P3 | `DataAlignmentValidator` | **删除** | 性价比低，用交易日历替代 |
| 新增 | 交易日历表 | **P1** | 用于校验交易日、排除节假日数据 |

## 四、关键代码修正示例

### 4.1 简化的 fill_stock

```python
def fill_stock(self, code: str, days: int = None, force: bool = False) -> int:
    days = days or self.config.days_back
    today = datetime.today().date()
    retention_start = today - timedelta(days=days)
    
    # 1. 确定填充范围（查DuckDB，不查SQLite状态表）
    fill_start, fill_end = self._get_fill_range(code, retention_start, today)
    if not fill_start:
        return 0  # 已是最新
    
    # 2. 标记运行中（简化状态）
    self.state.mark_running(code, fill_start, fill_end)
    
    try:
        # 3. 获取数据
        df = self.provider.get_daily_kline(code, fill_start, fill_end)
        if df.empty:
            self.state.mark_completed(code, 0, fill_end)
            return 0
        
        # 4. 数据校验（P0级）
        is_valid, errors = self.validator.validate(df)
        if not is_valid:
            raise DataValidationError(f"{code}: {errors}")
        
        # 5. 转换并写入
        records = convert_tushare_to_kline(df)
        self.store.insert_daily_klines(records)
        
        # 6. 更新状态（只记录latest_date）
        actual_latest = records['trade_date'].max()
        self.state.mark_completed(code, len(records), actual_latest)
        
        return len(records)
        
    except Exception as e:
        self.state.mark_failed(code, str(e))
        raise
```

2. **提升数据校验为P0**：写入前必须校验OHLC逻辑、停牌标记
3. **复权因子字段**：DuckDB表增加 `adj_factor`，移除 `adj_close`
