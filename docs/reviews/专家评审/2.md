## 二、潜在风险与改进建议

### 🔴 风险1：DuckDB日K线"只增不减"问题

**现状**：`INSERT OR REPLACE`保证单股票单日唯一，但历史数据理论上无限增长。

**你的方案**：5年保留 + 降采样归档。

**隐患**：
- 降采样为月K后，**回测精度下降**（无法做日线级回测）
- 用户可能期望"永久"保留日线数据

**建议**：
```python
# 提供两种模式供用户选择
DAILY_KLINE_MODE = {
    'standard': {'keep_years': 5, 'archive_monthly': True},   # 默认：降采样
    'premium': {'keep_years': 10, 'archive_monthly': False},   # 高配：全保留
    'minimal': {'keep_years': 2, 'archive_monthly': True},     # 低配：快速清理
}
```

或者**不真正删除**，而是**压缩归档到冷存储**：
```python
# 将超过5年的日K线导出为压缩Parquet，保留在archive/目录
# 查询时自动解压（透明访问）
def query_daily_kline(symbol, start_date, end_date):
    if start_date < five_years_ago:
        return query_from_archive(symbol, start_date, end_date)
    return query_from_duckdb(symbol, start_date, end_date)
```

### 🟡 风险2：SQLite的`price_history`与DuckDB日K线重复

**现状**：
- `price_history`：用户"查看过"的价格历史（SQLite，保留2年）
- `daily_kline`：全市场日K（DuckDB，保留5年）

**问题**：数据冗余，且`price_history`的**定义模糊**——"查看过"是前端行为，不应成为数据保留依据。

**建议**：
- **直接废弃`price_history`表**，查询需求走DuckDB的`daily_kline`
- 如果必须保留（比如缓存用户自选股的最新价格），改为**内存缓存**或**Redis**，不落地SQLite

### 🟡 风险3：JSON缓存的TTL执行机制未明确

**现状**：定义了TTL（如财务缓存7天），但未说明**如何检测过期**。

**建议**：
```python
# 在cache.py中实现基于mtime的清理
def cleanup_expired_cache():
    cache_dir = Path("data")
    for pattern, ttl_days in CACHE_RETENTION.items():
        for file in cache_dir.glob(f"{pattern}*.json"):
            age_days = (time.time() - file.stat().st_mtime) / 86400
            if age_days > ttl_days:
                file.unlink()
                logger.info(f"清理过期缓存: {file.name} (已存在{age_days:.1f}天)")
```

**优化**：对`market_*.json`（实时行情）用**1天TTL太保守**，建议改为**24小时滚动**或**按交易日清理**（避免周末积累）。

### 🟡 风险4：战力历史备份策略矛盾

**现状**：
- SQLite `cp_history`：永久保留
- JSON `cp_history.json`：30天
- 备份 `cp_history`：30天

**问题**：如果SQLite损坏，备份只有30天，**永久数据面临丢失风险**。

**建议**：
```python
BACKUP_RETENTION = {
    'sqlite_backup': 7,           # 短期快速恢复
    'cache_backup': 3,
    'cp_history_backup': 365,     # 战力历史备份改为1年（或永久）
    'critical_data_archive': None,  # 关键业务数据永久归档（月度快照）
}
```

或者实现**增量备份 + 云端同步**（可选功能）：
- 每月1日将`cp_history`导出为`cp_history_2026_04.parquet`
- 保留在`archive/critical/`目录，**永不删除**

### 🟢 建议5：存储监控的"健康度"算法

现状只有使用率阈值，建议增加**增长趋势预测**：

```python
def predict_storage_full():
    """基于近30天增长趋势，预测何时达到80%阈值"""
    history = get_daily_storage_usage(days=30)
    growth_rate = (history[-1] - history[0]) / 30  # GB/天
    
    if growth_rate <= 0:
        return "stable", None
    
    remaining = TOTAL_LIMIT * 0.8 - current_usage
    days_until_full = remaining / growth_rate
    
    if days_until_full < 30:
        return "critical", f"预计{days_until_full:.0f}天后触发清理"
    elif days_until_full < 90:
        return "warning", f"预计{days_until_full:.0f}天后触发清理"
    return "healthy", None
```

### 🟢 建议6：清理任务的幂等性

**场景**：如果23:30的清理任务执行到一半崩溃，如何保证**不重复清理、不遗漏**？

**建议**：
```python
def daily_cleanup():
    # 使用文件锁 + 状态记录
    lock_file = Path("data/.cleanup_lock")
    if lock_file.exists():
        logger.warning("清理任务已在运行，跳过")
        return
    
    try:
        lock_file.write_text(datetime.now().isoformat())
        
        # 记录本次清理范围（避免重复）
        state_file = Path("data/.cleanup_state")
        last_cleanup = json.loads(state_file.read_text()) if state_file.exists() else {}
        
        today = datetime.now().strftime("%Y-%m-%d")
        if last_cleanup.get('date') == today:
            logger.info("今日已清理，跳过")
            return
        
        # 执行清理...
        
        # 更新状态
        state_file.write_text(json.dumps({
            'date': today,
            'deleted_records': total_deleted,
            'freed_space': freed_mb
        }))
        
    finally:
        lock_file.unlink(missing_ok=True)
```
