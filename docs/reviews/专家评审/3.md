这份《TradeSnake 数据生命周期管理方案 v1.1》写得**非常出色**，具备直接落地到生产环境的质量。方案不仅结构清晰，而且展现了你对时序数据特性的深刻理解，特别是对“业务数据（战力/交易）”与“行情数据（K线/缓存）”的边界划分非常精准。
以下是我作为架构视角的详细评审意见，分为**亮点肯定**和**潜在隐患与优化建议**两部分：
---
### 一、 方案的四大亮点
1. **P0-P5 分级极其专业**
   明确提出 `cp_history`（战力历史）和用户资产数据“永不删除”，这是很多初级方案容易犯的错（为了省空间把历史特征洗掉，导致无法回测验证策略）。这个底线守得非常好。
2. **存储引擎的职责隔离很准确**
   SQLite 存强事务的业务数据，DuckDB 存重分析的时序K线，JSON 存易失性缓存。没有试图用一个大而全的数据库解决所有问题，避免了 IO 互相挤占。
3. **存储水位线保护机制（80%/95%）**
   引入了类似 Redis 的 `maxmemory-policy` 思想，在磁盘空间告急时不仅有告警，还有“缩短保留期50%”的降级策略。这对于桌面端软件防止“把用户C盘写爆”至关重要。
4. **实施路径规划合理**
   Phase 1 到 Phase 3 的迭代节奏很好，把最痛的点（DuckDB日K清理）放在 v18.5，把非强需求的 UI 配置化放在后面，符合敏捷开发原则。
---
### 二、 潜在隐患与优化建议（重点）
虽然方案大方向无懈可击，但在实际代码落地时，有几个细节容易“翻车”，建议在文档或代码设计中补齐：
#### 1. DuckDB 日K线“降采样归档”的业务逻辑断层
* **隐患：** 方案提到超过5年的日K降采样为月K后删除。但问题是：**如果用户在2年后想用5年前的日K线做精确回测怎么办？** 降采样成月K后，回测精度将大幅下降。
* **优化建议：** 
  不要把月K归档表当作回测的依赖，而是当作“看宏观大势”的展示用。
  必须在软件回测模块增加逻辑：**当回测区间超出本地日K保留期时，提示用户“历史数据精度不足，是否从远端拉取该片段数据用于本次回测？”**（即我上次提到的冷数据懒加载），拉取下来用完即弃，不进入核心 `daily_kline` 表。
#### 2. SQLite 每周 VACUUM 的“双刃剑”效应
* **隐患：** 方案中提到每周日凌晨执行 `vacuum_sqlite_if_needed()`。SQLite 的 `VACUUM` 是**全库重建**，过程中需要独占锁，且会生成一个临时文件（大小等于原数据库）。如果此时 `cp_history` 已经积累到几百MB，VACUUM 可能会导致软件卡死数秒，甚至因为瞬间磁盘空间翻倍而失败。
* **优化建议：**
  * 把“碎片整理”和“删除数据”解耦。
  * 删除 `price_history` 和 `alerts` 时，只是标记删除（产生碎片），**不要盲目全局 VACUUM**。
  * 改为：检查数据库的“碎片率”（可通过 `PRAGMA freelist_count` 估算），只有当碎片空间超过一定阈值（如 100MB）时，才在后台线程执行 VACUUM，并捕获异常防止阻塞主线程。
#### 3. `cp_history` 永久保留的“性能温水煮青蛙”
* **隐患：** 算一笔账：5000只股票 * 365天 = 182万条/年。5年后是900万条，10年后是1800万条。SQLite 处理千万级单表查询没问题，但如果有复杂的跨期排名计算（比如“计算某股票过去3年的战力历史分位数”），由于缺乏列式存储优势，可能会出现明显的慢查询。
* **优化建议：**
  * 目前不用改，但要在 `cp_history` 表设计里**预留分区字段或冷热标记**。
  * 建议加上 `is_hot BOOLEAN DEFAULT 1`。当数据超过2年时，标记为 `0`。查询默认带 `WHERE is_hot = 1`，避免全表扫描。未来的 v2.0 甚至可以把冷战力数据迁移到 DuckDB 中做列式分析。
#### 4. JSON 缓存清理的 I/O 效率
* **隐患：** 按文件后缀（`market_*.json`）清理，在 Windows 系统上遍历目录如果遇到大量小文件（比如上万个 Tushare 缓存碎片），`os.scandir` 会比较慢。
* **优化建议：** 
  * 不要信任文件的修改时间，因为 Git 拉取或文件同步软件会改变 mtime。
  * 建议在 JSON 文件内部写入一个 `{"expire_at": 1712444800, "data": {...}}` 字段。清理时采用**“清理时淘汰”**策略：程序读取缓存时如果发现 `expire_at` 过期，顺手删除该文件；定时任务仅负责扫描清理长期未访问的孤儿文件。
#### 5. 备份与清理的时序安全
* **隐患：** 方案中“每日 23:30 随备份一起执行清理”。如果先执行了清理，再执行备份，那备份出来的就是“已经被删过数据”的库，失去了备份防范误删的意义。
* **优化建议：** 严格执行 **先备份，后清理**。流程应为：
  `1. 暂停数据写入 -> 2. 拷贝 DB 文件至 backup -> 3. 校验备份文件完整性 -> 4. 执行 DELETE 清理 -> 5. 恢复写入`。

它对 TradeSnake 当前阶段的数据痛点抓得极准。我上述提出的 5 点优化建议，属于“防微杜渐”的高级架构考量，**不需要在 v18.5 全部实现**，但建议你作为 Tech Debt（技术债）记录在案的 TODO 里：
