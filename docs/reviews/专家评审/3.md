### 一、 架构与逻辑层面的改进建议
#### 1. GapDetector（缺口检测）的致命漏洞：缺乏“交易日历”
**问题**：当前的缺口检测逻辑如果是基于自然日（`latest_date + 1`）去比对，会把周末、法定节假日误判为“数据缺口”，导致触发大量无效的 API 请求，甚至可能引发 Tushare 限速封禁。
**建议**：
*   **必须引入交易日历表**（`trade_cal`）。在 DuckDB 中维护一张交易日历表。
*   `detect_gaps` 的逻辑应该是：获取数据库最新日期 -> 查询交易日历找到下一个实际交易日 -> 如果该交易日 < 当前日期，则判定为缺口。
#### 2. API 同步阻塞风险
**问题**：第六章 API 接口设计中，`POST /data/fill` 如果触发 `fill_all(limit=200, days=730)`，按照 `rate_limit_sleep = 0.5` 秒计算，200只股票至少需要 100 秒。这会导致 HTTP 请求超时。
**建议**：
*   API 层应该改为**异步任务模式**。接口只负责创建任务，返回 `task_id`，然后使用后台线程（如 Python 的 `threading`）或任务队列（如 Celery/ARQ）去执行 `KlineFiller`。
*   前端通过 `GET /data/fill/status?task_id=xxx` 轮询进度。
#### 3. FillerState (SQLite) 的并发锁问题
**问题**：SQLite 不擅长高并发写入。如果通过 API 同时触发多个单股填充任务，`mark_running` 可能会触发数据库锁。
**建议**：
*   在 `FillerState` 的数据库连接字符串中加上超时和 WAL 模式：`sqlite:///tradesnake.db?timeout=10&isolation_level=None`，并在初始化时执行 `PRAGMA journal_mode=WAL;`。
*   增加一个**僵尸进程回收机制**：如果某条记录 `status='running'` 但 `updated_at` 超过 2 小时，应自动重置为 `failed`，防止程序崩溃后死锁。

### 二、 金融业务细节的修正
#### 1. 复权因子的“级联更新”问题（核心痛点）
**问题**：文档中设计了 `adj_factor` 字段，但遗漏了复权计算的一个核心特性：**历史复权因子是会变的**。
假设某股票今天（4月10日）进行了“10送10”，那么它去年所有的 `adj_factor` 都会被 Tushare 重新计算并更新。如果你只在第一次填充时存了 `adj_factor`，今天之后你看去年的 K 线图就会断层。
**建议**：
*   在 `update_scheduler` 每日盘后拉取数据时，除了拉取今天的K线，**必须强制拉取全量的 `adj_factor` 表数据并覆盖更新**。Tushare 的 `daily` 接口自带 `adj_factor`，但最好单独拉取 `adj_factor` 接口以确保准确。
#### 2. DataValidator（数据校验）的涨跌幅硬编码
**问题**：`-10% <= change_pct <= 10%` 这个规则过于死板。
*   科创板/创业板（注册制）是 ±20%。
*   北交所是 ±30%。
*   新股上市前5天无涨跌幅限制。
*   ST 股票是 ±5%。
**建议**：
*   不要把涨跌幅超限作为硬性阻断错误（不要直接判为 invalid）。
*   应该改为**告警**机制：如果超出常规的 ±10%，打上 `is_abnormal=True` 的标签，或者只记录日志。真正的异常（如 OHLC 逻辑错误）才应该阻断入库。
#### 3. Tushare 停牌数据的坑
**问题**：Tushare 对停牌股票的处理比较奇葩：有时候返回当天的一行数据（开盘=收盘=最高=最低=昨收，成交量=0），有时候**根本不返回这一天的数据**。
**建议**：
*   在 `convert_tushare_to_kline` 中，如果发现拉取的日期段在交易日历中存在，但 Tushare 返回的 DF 中缺失，应该在本地**主动补齐**这一天的记录，填充为 `{open: NaN, high: NaN, low: NaN, close: NaN, volume: 0, is_suspended: True}`。这对于后续时间序列分析（如计算 MA 均线）至关重要，否则 pandas 的 rolling 操作会错位。

### 三、 代码实现层面的优化
#### 1. `convert_tushare_to_kline` 遗漏了复权因子
文档 4.4 节的转换函数中，没有包含 `adj_factor` 的处理。
```python
# 建议补充：
result['adj_factor'] = df['adj_factor'].astype(float) 
# Tushare直接返回当日的复权因子，无需额外计算
```
#### 2. 多源备份的数据格式对齐
**问题**：`MultiSourceProvider` 假设所有备用源都返回一样的 `List[Dict]`，但实际上：
*   Tushare 返回 `000001.SZ`，BaoStock 返回 `sz.000001`，AkShare 返回 `000001`。
*   Tushare 成交量是“手”，BaoStock 成交量可能是“股”。
**建议**：
*   每个 Provider 内部必须自己做标准化，统一输出为 `KlineRecord` Pydantic Model 或标准的 DataFrame 格式后，再交给 `MultiSourceProvider` 返回。不要在 Filler 层做兼容。
#### 3. `fill_all` 的限速逻辑优化
```python
# 当前逻辑：
for stock in stocks:
    # ... 填充逻辑 ...
    time.sleep(self.config.rate_limit_sleep) # 即使失败也sleep？
```
**建议**：只有成功获取数据时才 sleep。如果网络报错抛异常重试，不应该浪费限速时间。
