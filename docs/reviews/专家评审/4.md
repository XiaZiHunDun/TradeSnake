这份策略优化设计文档结构清晰、目标明确，整体思路可行。但从严谨性和可落地角度，有几个关键问题值得深入探讨。

---

## 一、核心问题

**1. 策略定位与参数冲突**

文档定义"稳健型策略"，却设置**最大持仓天数5天+止损-10%**。这实质是**短线交易型**策略，与"稳健"二字存在矛盾。短线策略天然波动大、胜率不稳定，强制要求"最大回撤≤15%"可能过于严苛，会导致参数选择空间被过度压缩。

**再审视持仓集中度约束**：单只仓位8-15%，持仓5-8只，意味着总仓位在**40%-120%之间浮动**。这是一个隐含的杠杆/仓位管理问题，但文档完全未定义满仓规则——是等权分配？还是按信号强弱分配？这会直接影响回测结果可比性。

**2. 战力公式权重之和为95%而非100%**

growth(30%) + value(25%) + quality(20%) + momentum(8%) + real_time(2%) + risk_penalty(10%) = **95%**。少掉的5%是设计如此还是遗漏？如果是遗漏，所有基于v19.6的回测基线都需重算。

**3. 数据划分存在信息泄露风险**

阶段2验证方式为：训练集2024年数据、测试集2025年数据、最终评估2024+2025全区间。

问题在于：如果先在全区间上做完阶段1策略对比，选出最优策略再进入阶段2，那么**阶段2测试集就不再是真正的预留样本**——因为策略选择本身参考了2025年数据。正确流程应是：阶段1仅在训练集(2024)上对比 → 选出最优策略 → 阶段2同样仅在训练集(2024)上参数扫描 → 最终一次性在测试集(2025)上评估。

**4. 融合公式中的子分数未定义来源**

```
fused_score = 0.4 * cp_score + 0.35 * gain_score + 0.25 * prob_score
```

`cp_score` 是战力总分？`gain_score` 是 `predicted_gain_5d`？`prob_score` 是 `up_probability_5d`？这些子分数如何归一化到可加权区间？如果不归一化，量纲差异会使权重形同虚设。

## 二、次要问题

**5. 最小交易金额5万门槛逻辑不清**

初始资金100万，单笔最小交易5万意味着单票最低仓位约16.7%，与"单只8-15%"的约束存在逻辑矛盾——选8只时每只12.5%仓位仅7.5万，勉强够；但选5只时每只最多15%仓位对应的实际金额可能远超5万下限。

**6. 缺少交易成本假设**

两日历年数据的短线高频策略（持仓≤5天），如果没有滑点、佣金、印花税的成本模型，回测收益会被显著高估。A股印花税+佣金双边约0.13%，对于平均持仓5天的策略，年化磨损约5-6%——不是小数。

**7. 因子贡献分析中的风险因子处理**

因子分析将 `risk_penalty` 与其他因子并列。但看战力公式，risk_penalty是**惩罚项**（已乘以10%权重，且在融合公式外生效），其贡献分析需要在正确的作用路径上评估，而非与其他正向因子简单并列做逐步剔除。

**8. 缺少缺失值处理规则**

2024-2025年期间有IPO、ST戴帽摘帽、停复牌等情形。持仓5天的策略遇到停牌如何处理？如果直接跳过，回溯结果会有幸存者偏差；如果按停牌前价格持有，又与实际可交易性不符。

## 三、改进建议

**短期（M1前）**：
- 明确总仓位规则（满仓等权 or 仓位与信号强度挂钩）
- 补全战力公式权重到100%或给出说明
- 严格定义训练/测试数据集隔离策略
- 添加交易成本模型（至少含印花税+佣金+滑点0.1%）

**中期（M2阶段）**：
- 将参数扫描的"最大回撤≤15%"改为软约束（设置为过滤条件的较高优先级而非硬性排除，因为不同行情下同类策略的回撤分布差异大）
- 对持仓天数与止损阈值做交叉验证，而非独立扫描（这两者强相关：持仓3天配-5%止损 vs 持仓10天配-15%止损是不同的逻辑）
- 加入收益率波动/最大回撤持续时间的稳定性指标，补充单纯看"胜率"的不足

**长期（M3+）**：
- 因子分析改用 Shapley值或边际效应分解，避免逐步剔除法对因子相关性的敏感
- 设计一个基线对照（如等权持有沪深300成分股）来横向对比策略超额收益的显著性
- 增加Walk-forward验证，而非单一训练/测试切分

---

**总体评价**：方案方向正确，但量化细节（数据划分、仓位规则、成本模型、因子归一化）上的疏漏可能导致结论不可靠。建议先补齐这些基础规范再进入编码阶段，而不是在代码中"顺便解决"。
