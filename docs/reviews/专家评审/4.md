**问题**：DuckDB日K线清理和SQLite清理不在同一事务，可能导致：
- 清理后用户查询历史数据时，部分数据已删除但关联索引未更新
- 极端情况下清理失败，数据状态不一致

**建议补充**：
```python
def safe_cleanup_with_transaction():
    """带事务保护的清理"""
    # 1. 记录清理前的数据快照（用于回滚）
    cleanup_marker = {
        'timestamp': datetime.now().isoformat(),
        'daily_kline_cutoff': cutoff_date,

    }
    
    try:
        # 2. 执行清理
```

### 2. 大批量删除的IO压力

**问题**：日K线5年数据可能涉及数百万条记录，一次性DELETE可能造成：
- DuckDB写入放大
- 磁盘IO阻塞
- 影响服务响应

**建议**：分批删除
```python
def batch_cleanup_daily_kline(keep_years: int = 5, batch_size: int = 10000):
    """分批清理，避免IO阻塞"""
    cutoff = datetime.now() - timedelta(days=keep_years * 365)
    
    while True:
        deleted = conn.execute("""
            DELETE FROM daily_kline 
            WHERE trade_date < ? 
            LIMIT ?
        """, [cutoff.isoformat(), batch_size])
        
        if deleted == 0:
            break
        
        # 每批之间短暂休眠，避免长时间阻塞
        time.sleep(0.1)
        
        log(f"已清理 {deleted} 条记录")
```

### 3. 清理后索引碎片化

**问题**：大量DELETE后，SQLite和DuckDB会产生索引碎片，影响查询性能。

**建议**：增加定期 VACUUM
```python
def weekly_vacuum():
    """每周日凌晨清理后执行"""
    # SQLite VACUUM
    conn.execute("VACUUM")
    
    # DuckDB 可通过重新创建索引优化
    conn.execute("ALTER TABLE daily_kline REORGANIZE")
```
