## 一、主要优点

1. **分级清晰（P0~P5）**：明确区分了“永不删除”的用户资产/战力历史与可清理的缓存、K线数据，避免误删风险。
2. **存储架构完整**：同时管理 SQLite（业务数据）、DuckDB（K线）、JSON 缓存和备份，符合实际项目需求。
3. **渐进式清理 + 存储上限保护**：分批删除、阈值触发，能有效防止 IO 卡顿和磁盘写满。
4. **用户可配置**：保留天数、清理开关等设计，提升了软件的灵活性。
5. **战力历史（cp_history）单独强调**：作为核心业务数据永久保留，理由充分（数据量可控、支持长期回测）。

## 二、存在的问题与潜在风险

### 1. **DuckDB 日K线“降采样归档”实现不完整**
- 方案中提到“超过5年的日K线降采样为月K后删除”，但：
  - 未明确归档表（`monthly_kline_archive`）的存储位置（同一个 DuckDB 文件？另建文件？）。
  - 未说明查询时如何透明地合并日K + 月K（用户请求5年以上数据时需跨表查询）。
  - 如果用户选择“永久保留”日K，降采样策略应跳过。

**建议**：  
- 将归档月K存入单独的 DuckDB 文件（如 `archive.duckdb`），避免主库膨胀。  
- 在查询层封装一个视图或函数，根据时间范围自动路由：`trade_date < 5年前` 查归档表，否则查日K表。

### 2. **SQLite 历史清理缺少 VACUUM 的触发条件细节**
- 每周执行 `vacuum_sqlite_if_needed()`，但未定义“需要”的判断标准（如空闲空间 > 20% 或删除行数 > 1000）。频繁 VACUUM 会重写整个数据库文件，可能造成长时间锁表。

**建议**：  
- 增加条件：仅当删除的记录数超过总行数的 10% 或数据库文件空闲页超过 20% 时，才在低峰期（如凌晨3点）执行 `VACUUM`。  
- 或者使用 `VACUUM INTO` 创建新文件再替换（SQLite 3.27+），减少锁定时间。

### 3. **JSON 缓存清理策略过于静态**
- 所有缓存按固定 TTL 删除，但未考虑“最后访问时间”（LRU）。例如 `stock_pool.json` 虽然 TTL=7天，但用户可能每天都修改，直接删除会丢失用户未保存的配置。

**建议**：  
- 区分“可重新下载的缓存”（如行情、财务）与“用户生成/修改的配置”（如股票池）。  
- 对于用户修改过的文件，应改为“更新时覆盖”，而不是基于 TTL 自动删除。可以在元数据中记录 `user_modified = true`，跳过自动清理。

### 4. **清理任务与备份任务的时序依赖未明确**
- 方案中说“每日收盘后23:30执行清理，然后备份”。但若清理失败或耗时过长，可能影响备份窗口。另外，备份中是否包含正在被清理的表（如 DuckDB）？

**建议**：  
- 调整顺序：**先备份，后清理**。这样即使清理逻辑出错，备份仍保留完整旧数据。  
- 备份时对 SQLite 使用 `BEGIN IMMEDIATE` 或备份 API（`backup`），对 DuckDB 使用 `COPY TO` 避免锁竞争。

### 5. **存储上限保护中的“强制清理至80%”可能过度**
- 当使用率 >95% 时，方案要求“强制清理至80%”，这可能导致大量数据被过早删除（例如将5年的日K直接砍到1年）。用户可能无法接受。

**建议**：  
- 改为“触发用户确认”模式：弹窗告知存储不足，让用户选择清理策略（缩短保留期 / 删除旧备份 / 手动扩容）。  
- 自动清理时，优先删除临时文件和最旧的缓存，最后才缩短 K 线保留期，并记录日志。

### 6. **缺少对“数据空洞”和“部分清理失败”的容错处理**
- 例如清理分钟K线时中途程序崩溃，可能导致某些股票的数据保留不全。下次清理时可能重复删除已删除的数据（幂等性还好），但用户查询时可能遇到缺失日期。

**建议**：  
- 在清理事务中（DuckDB 支持事务）批量删除，或记录清理检查点（如 `last_cleanup_date`）。  
- 对于 SQLite，使用 `DELETE ... WHERE ...` 本身是原子的，但跨多表需要显式事务。

### 7. **性能考虑不足**
- 方案中每次清理都扫描全表（如 `DELETE FROM daily_kline WHERE trade_date < ...`），在 DuckDB 中这很快，但若 `daily_kline` 达到600万行，删除操作仍会产生大量写放大。
- 每周对 SQLite 执行 `DELETE FROM price_history WHERE ...`，如果表很大且没有合适索引，会锁表很长时间。

**建议**：  
- 对 `trade_date` 建立索引（DuckDB 中可按时间分区）。  
- 采用“分批次删除”：每次只删 10000 行，循环直到删完，避免长事务。  
- 考虑使用“滑动窗口表”设计：例如分钟K线按月分表（`minute_kline_202501`），直接 `DROP TABLE` 旧月份表，比逐行删除高效得多。

## 三、遗漏项或可增强点

### 1. **未考虑网络同步场景**
- 如果用户有多设备或云端同步，本地清理后云端数据是否保留？应明确清理是否影响同步。

### 2. **缺少清理操作的审计日志**
- 建议记录每次清理的时间、清理类型、删除记录数、释放空间大小，便于排查问题。

### 3. **战力历史（cp_history）虽然永久保留，但未设置行数上限**
- 虽然预估可控，但若用户每日全量计算5000只股票的战力，5年数据量约为 5000×1825 ≈ 912万行。SQLite 单表超过千万行后查询性能可能下降。

**建议**：  
- 按年分表（`cp_history_2025`），或者增加自动归档策略（例如将超过3年的数据迁移到只读的归档库，但保留查询能力）。

### 4. **清理调度频率与具体时间点可优化**
- 每日23:30可能恰好是某些用户交易后复盘的时间，建议改为凌晨2:00~4:00。
- 周清理和日清理可能重叠，应合并或错开。

## 四、实施优先级建议

| 优先级 | 改进项 | 原因 |
|-------|--------|------|
| **高** | 实现 DuckDB 日K线降采样并完善查询路由 | 当前方案不完整，会导致用户查询5年前数据失败 |
| **高** | 调整清理与备份的顺序（先备份后清理） | 避免清理错误导致数据无法恢复 |
| **高** | 增加存储上限保护的用户确认机制 | 防止强制清理引起用户不满 |
| **中** | 优化删除方式（分批、按月分表） | 保证大表清理时软件仍可响应 |
| **中** | 增加清理审计日志 | 便于后期运维 |
| **低** | 为 cp_history 设计分表或归档 | 长期数据量增长后优化 |
