## 🔴 严重问题（影响资金安全！必须修复）

### 1. 【致命】异步代码中使用了同步锁 ⚠️⚠️⚠️

**文件**：`backend/api/routes.py:5, 22`

```python
import threading      # 问题！
...
_cp_lock = threading.Lock()  # 这是同步锁！

@router.get("/api/cp/top")
async def get_cp_top(...):
    # 这里用的是 async def，但使用的是 threading.Lock！
    # threading.Lock 会阻塞整个事件循环！
```

**问题分析**：
- FastAPI 是异步框架，所有路由函数都是 `async def`
- `threading.Lock` 在 async 函数中调用时，会**阻塞整个事件循环**
- 这意味着当一个请求持有锁时，其他所有请求都在等待
- 在高并发场景下，会导致请求堆积、响应变慢，甚至**死锁**！

**风险等级**：极高 - 可能导致系统崩溃

**修复方案**：
```python
import asyncio

_cp_lock = asyncio.Lock()  # 使用异步锁！

# 使用方式不变，with 语句兼容
async def refresh_cp_data(...):
    async with _cp_lock:
        ...
```

**影响**：
- 当前锁范围很小（只在 refresh_cp_data 内部），影响有限
- 但如果以后扩展锁的范围，问题会急剧放大
- 生产环境高并发时必定出问题

### 2. 【严重】历史数据文件无保护 ⚠️⚠️

**文件**：`backend/core/history.py:40-46`

```python
def save_history(stocks: List[Dict], date: str = None):
    ...
    try:
        with open(HISTORY_FILE, 'w', encoding='utf-8') as f:
            json.dump(history, f, ensure_ascii=False, indent=2)
```

**问题分析**：
- 直接写入文件，无原子性保证
- 如果写入过程中断（如进程被kill），文件会损坏
- 多线程/多进程同时写入会导致数据丢失
- 没有备份机制，无法恢复

**风险等级**：高 - 历史数据是核心资产，一旦丢失无法恢复

**修复方案**：
```python
import tempfile
import shutil

def save_history_safe(stocks: List[Dict], date: str = None):
    """原子写入 + 备份"""
    ensure_dir()

    if date is None:
        date = datetime.now().strftime("%Y-%m-%d")

    history = load_history()

    # 构建新数据
    history[date] = {
        "stocks": {s["code"]: s for s in stocks},
        "saved_at": datetime.now().isoformat()
    }

    # 清理旧数据
    dates = sorted(history.keys(), reverse=True)
    if len(dates) > 30:
        for old_date in dates[30:]:
            del history[old_date]

    # 原子写入：先写临时文件，再 rename
    temp_file = tempfile.NamedTemporaryFile(
        mode='w',
        dir=HISTORY_DIR,
        delete=False,
        suffix='.tmp'
    )
    try:
        json.dump(history, temp_file, ensure_ascii=False, indent=2)
        temp_file.close()
        # 原子替换
        shutil.move(temp_file.name, HISTORY_FILE)
        return True
    except Exception as e:
        # 清理临时文件
        if os.path.exists(temp_file.name):
            os.unlink(temp_file.name)
        print(f"保存历史记录失败: {e}")
        return False
```

## 🟠 重要问题（可能影响收益）

### 3. 【重要】数据源切换无熔断机制

**文件**：`backend/data/fetcher.py:182-194`

```python
data = self._fetch_from_tencent(codes)
if data:
    return data
# 腾讯失败，尝试新浪
data = self._fetch_from_sina(codes)
```

**问题分析**：
- 如果腾讯连续失败，每次请求都会重试 3 次
- 这会浪费大量时间在无效请求上
- 在腾讯恢复之前，每个请求都会经历 3 次重试延迟
- 没有"快速失败"机制

**修复建议**：
```python
class MarketDataFetcher:
    def __init__(self):
        ...
        self._tencent_failures = 0
        self._sina_failures = 0
        self._circuit_breaker_threshold = 5  # 连续失败5次后熔断
        self._circuit_breaker_cooldown = 60  # 60秒后恢复

    def get_market_data(self, codes: List[str], use_cache: bool = True) -> List[Dict]:
        ...

        # 检查熔断状态
        if self._tencent_failures >= self._circuit_breaker_threshold:
            # 直接跳过腾讯，尝试新浪
            data = self._fetch_from_sina(codes)
            if data:
                self._tencent_failures = 0
            return data

        data = self._fetch_from_tencent(codes)
        if data:
            self._tencent_failures = 0  # 成功后重置计数
            return data

        # 腾讯失败
        self._tencent_failures += 1
        ...
```

### 5. 【中等】缓存无容量限制

**文件**：`backend/data/fetcher.py`

```python
cache_key = md5_hash(','.join(sorted(codes)))
write_cache(f'market_{cache_key}', data, ...)
```

**问题**：不同股票组合会产生无数不同的缓存文件，可能占满磁盘空间。

**修复建议**：
- 限制缓存文件总数（如 LRU，保留最近 100 个）
- 或使用内存缓存 + 定期持久化
