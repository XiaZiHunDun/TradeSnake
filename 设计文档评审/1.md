# 股市贪吃蛇 (TradeSnake) 设计文档评审报告
## 一、整体评价
这份设计文档**框架完整、核心概念创新且落地路径清晰**，将股票投资与贪吃蛇游戏做了深度的场景映射，既保留了投资分析的专业性，又通过游戏化降低了操作门槛，作为私人工具的初始设计具备很强的可行性。

文档的优势集中在：
1. 核心逻辑自洽：战力值体系贯穿所有功能模块，贪吃蛇的游戏隐喻与股票操作的映射关系明确；
2. 版本规划合理：v1.0聚焦核心功能（战力面板、榜单、查询），剥离高复杂度模块（实时行情、市场地图），符合「先跑通再优化」的开发原则；
3. 体验设计具象：UI/交互细节（如蛇身卡片、战力段位、视觉反馈）有具体落地参考，避免了抽象化设计；
4. 技术选型务实：基于私人工具的定位选择Python/FastAPI/React/SQLite，无过度技术设计，开发成本可控。

## 二、核心亮点
### 1. 价值体系创新
摒弃传统「以金额论成败」的投资视角，用「战力值」重构投资决策逻辑，既规避了短期价格波动的干扰，又通过「置换」而非「买卖」的操作定义，引导长期价值判断，贴合价值投资本质。

### 2. 游戏化设计落地性强
- 视觉元素（段位系统、蛇身持仓、战力能量条）既保留游戏趣味性，又不偏离股票分析的核心；
- 交互反馈（战力升降动画、置换交换效果）能直观传递投资结果，降低数据解读成本；
- 操作映射（加速=重仓、减速=轻仓）符合用户直觉，无认知断层。

### 3. 战力公式设计务实
v1.0简化公式（成长分40%+价值分30%+趋势分30%）兼顾「基本面+技术面」，且支持参数可调，既满足初期跑通需求，又为后续个性化优化预留空间；标准化采用「A股百分位排名」，避免了因子绝对值波动的干扰。

## 三、待优化/补充的关键项
### 1. 战力计算模块
#### 待补充点
- **因子异常值处理细则**：文档仅提及「PE为负/过高取低值」，但未定义「过高」的阈值（如PE>100？）、净利润增速为负时的计分规则（直接归0？反向计分？）；
- **标准化边界条件**：新股（上市<1年）用营收增速替代成长分，但未说明营收增速的计算周期（单季度？年度？），且未考虑「营收增速为负」的情况；
- **数据更新逻辑**：日更数据的具体更新节点（收盘后几点？）、数据缺失/接口异常时的兜底方案（用前一日数据？标记「数据异常」？）未明确。

#### 建议
```
# 补充异常值处理示例（可纳入v1.0设计）
成长分异常处理：
- 净利润增速 < 0 → 成长分 = 0
- 营收增速（新股）< 0 → 成长分 = 0
价值分异常处理：
- PE < 0 → 价值分 = 0
- PE > 80 → 价值分 = （80/实际PE）× 基础价值分（避免极端值拉低整体）
- ROE < 0 → 价值分 = 0
标准化规则：
- 所有因子先按行业分类排名（而非全A股），再计算百分位（如消费股PE与消费股比，而非与周期股比）
```

### 2. 功能模块落地细节
#### 待补充点
- **持仓管理**：手动录入仅提及「名称/代码/持股数/成本价」，但成本价是否参与战力计算？文档中个人战力=Σ(持股数×股票战力值)，成本价仅用于浮盈展示？需明确；
- **战力榜单**：仅提TOP50，但未说明榜单筛选范围（全A股？沪深300？）、更新频率（日更几点？）、榜单排序规则（仅按战力值？是否区分行业？）；
- **置换建议**：v1.0虽不做自动建议，但「单股战力对比」需明确对比维度（仅总战力值？还是拆分成长/价值/趋势分对比？）。

#### 建议
- 成本价仅作为「参考信息」，不纳入战力计算（避免成本价干扰战力核心逻辑）；
- 战力榜单默认展示「全A股战力TOP50」，新增「行业战力榜」筛选项（如消费/新能源/半导体），贴合「同板块置换」的核心逻辑；
- 单股战力对比页拆分「因子得分雷达图」，直观展示「该股票哪类因子弱于竞品」。

### 3. 技术落地细节
#### 待补充点
- **数据源具体对接方式**：东方财富/同花顺的API是否有免费额度？本地数据获取是爬取还是官方API？爬取的合规性/稳定性需评估；
- **存储设计**：SQLite需明确表结构（如股票基础信息表、战力值历史表、持仓表），避免后期数据冗余；
- **战力趋势计算**：近7日/30日战力走势的「战力值」是取每日收盘后计算值？还是实时计算（v1.0日更则无需）？

#### 建议
- 初期优先用「Tushare（免费版）+ 东方财富网页爬取」组合获取数据（财报+行情），降低API接入成本；
- 补充核心表结构示例：
```sql
-- 股票基础信息表
CREATE TABLE stocks (
    code TEXT PRIMARY KEY,  -- 股票代码
    name TEXT,              -- 股票名称
    industry TEXT,          -- 所属行业
    list_date TEXT          -- 上市日期
);

-- 战力值历史表
CREATE TABLE stock_cp (
    code TEXT,
    date TEXT,              -- 计算日期
    growth_score INTEGER,   -- 成长分(0-100)
    value_score INTEGER,    -- 价值分(0-100)
    momentum_score INTEGER, -- 趋势分(0-100)
    total_cp INTEGER,       -- 总战力值
    PRIMARY KEY (code, date)
);

-- 个人持仓表
CREATE TABLE holdings (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    code TEXT,
    quantity INTEGER,       -- 持股数
    cost_price REAL,        -- 成本价
    update_date TEXT        -- 最后更新日期
);
```

### 4. 体验设计补充
#### 待优化点
- **段位系统**：战力范围（青铜0-1999、白银2000-4999）未说明「战力值上限」的设定逻辑（如王者30000+是否合理？），且未考虑「持仓规模差异」（100股茅台 vs 10000股茅台的战力值差距过大，段位是否应做「人均战力」归一化？）；
- **视觉反馈**：仅提及动画效果，但未考虑「低视觉刺激场景」（如夜间模式），需补充深色模式下的配色适配（如战力高的绿色避免刺眼）；
- **操作流程**：「加入观察/模拟买入」在v1.0是否支持？需明确v1.0仅支持「真实持仓录入」还是「模拟持仓」。

#### 建议
- 段位系统新增「人均战力」维度（总战力/持股总数），作为「段位判定辅助指标」，避免单纯靠持仓数量堆战力；
- 补充深色模式配色调整：
  ```
  深色模式适配：
  战力高: #00e070 (柔和绿，替代电光绿)
  战力中: #e6c000 (柔和黄，替代柠檬黄)
  战力低: #e64a4a (柔和红，替代警示红)
  ```
- v1.0支持「真实持仓+模拟持仓」双模式，模拟持仓用于测试「置换效果」，不影响总战力。

## 四、开发优先级建议
结合「私人工具」的核心诉求，建议v1.0开发优先级排序：
1. **战力计算引擎**（核心基础，无战力值则其他功能无数据支撑）；
2. **个人战力面板**（核心体验，优先实现总战力、持仓明细、战力趋势）；
3. **单股战力查询**（满足「个股分析」核心需求）；
4. **战力榜单**（补充市场视角，优先级略低于前三者）；
5. 游戏化UI（先实现核心数字展示，动画/特效可后期迭代）。

## 五、风险提示
1. **数据准确性风险**：免费数据源（如Tushare、爬取）可能存在数据延迟/错误，需在UI中标注「数据仅供参考」，且预留手动修正战力值的入口；
2. **过度游戏化风险**：需避免「段位升级」「战力排名」带来的投机心理，可在UI中增加「投资有风险」提示，且战力值仅作为分析参考，不构成投资建议；
3. **维护成本风险**：因子规则、数据源可能随时间变化，需设计「配置文件化」的战力公式（如用YAML配置因子权重/异常值规则），避免硬编码。

## 六、总结
这份设计文档已具备「可落地的MVP版本」核心要素，仅需补充「战力计算细则、存储设计、边界条件处理」等细节，即可进入开发阶段。核心建议是：
- 开发阶段严格遵循「v1.0功能清单」，不额外增加复杂度；
- 优先验证「战力值计算逻辑」的合理性（可先手动计算几只股票的战力值，确认符合直觉）；
- 游戏化UI先实现「核心视觉元素」（战力数字、段位、持仓蛇身），动画/特效可后期迭代。

整体而言，该设计既保留了投资分析的专业性，又通过游戏化降低了操作门槛，作为私人工具具备很高的实用价值，落地后可有效辅助价值投资决策。
